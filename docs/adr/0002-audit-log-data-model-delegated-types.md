# 0002_監査ログのデータモデル（汎用 AuditLog + Delegated Types）

## ステータス

承認済み

## 日付

2026-06-08

## 信頼度

中〜高 — スパム以外の管理操作監査ログ（キーワード編集、スパマー登録変更、ユーザーBAN、権限変更等）を追加する具体的な予定があることをユーザーが確認済みで、汎用化と型別テーブルの前提が成立する。`delegated_type` は Rails 6.1 以降の安定 API（本プロジェクト 7.2.3.1）。一方、追加種別の実装はこれからであり、運用での検証はこれから。

## 再評価条件

- 予定していた他の監査ログ種別が長期間追加されず、スパムが唯一の型に留まった場合（汎用層が単一用途への過剰構造になる。撤去は容易だが、設計判断としては見直し対象）。
- 監査ログの生成頻度が極端に高くなり、1操作あたり2行（superclass + subtype）の書き込みコストが問題化した場合。

## コンテキスト

管理操作の監査ログを新設する（初回はスパム手動認定の記録/取消）。これは「管理機能の監査ログの一種」であり、スパム以外にも複数の監査対象を追加する具体的な予定がある。

### 現状の問題点

- 監査ログの基盤が存在しない（`ProjectAccessLog` はアクセス解析用で監査用途ではない）。
- 監査イベントは種別ごとにペイロードが異なる（スパム＝対象＋操作種別、設定変更＝前後値、BAN＝理由＋期間など）。一方、操作者・操作日時といったメタデータは全種別で共通する。
- 操作者横断の監査（「管理者Xの全操作」）や共通の閲覧UIを将来提供するには、種別をまたぐ共通の問い合わせ口が要る。

### 制約条件

- 各監査種別の型別項目は、DBレベルの整合性（NOT NULL、外部キー、インデックス）を確保したい。
- Rails 7.2 系。`delegated_type`・STI・ポリモーフィックいずれも利用可能。

## 決定

汎用の `AuditLog` を superclass とし、Rails の `delegated_type` で種別別モデルに委譲する構造を採用する。スパム認定監査（`SpamModerationAudit`）を最初の委譲先とする。

### 実装方針

1. 共通テーブル `audit_logs` に、全監査種別で共通のメタデータ（操作者、委譲参照 `auditable_type`/`auditable_id`、発生日時）を持たせる。`AuditLog` は `belongs_to :operator`（User）と `delegated_type :auditable, types: %w[SpamModerationAudit]` を定義する。
2. 種別別テーブル `spam_moderation_audits` に、スパム固有の項目（操作種別＝記録/取消、対象種別、対象ID）を持たせる。型別項目には適切な NOT NULL・インデックスを付与する。
3. 種別別モデルは共通の `Auditable` 責務（`has_one :audit_log, as: :auditable`）を持ち、`AuditLog` 経由で操作者・日時にアクセスする。
4. 監査ログの生成は、対象の状態変更と同一トランザクション内で行い、superclass と subtype を原子的に作成する（ADR-0001 の操作者捕捉と組み合わせ、`operator` は `Current.admin`）。
5. 新しい監査種別の追加は、種別別テーブル/モデルの新設と `delegated_type` の `types` への追加で行う。

## 結果

### ポジティブな影響

1. **拡張可能な監査基盤**
   - 共通メタデータ・共通クエリ（操作者別・期間別）・共通閲覧UIを全監査種別で共有できる。新種別は型別テーブルとモデルの追加で対応できる。

2. **型別項目の整合性**
   - 各種別が独自テーブルを持つため、型別項目に NOT NULL・外部キー・インデックスを付与できる。スパース列（STI）やスキーマレスなJSON（ポリモーフィック＋payload）の弱点を避けられる。

3. **型別の振る舞いの局所化**
   - 種別固有のロジック・バリデーションを各 subtype モデルに閉じ込められる。

### ネガティブな影響・トレードオフ

1. **初期構造の複雑化**
   - 最初の1種（スパム）に対して superclass + subtype の2テーブルと `delegated_type` 配線が必要。
   - 対策: 他種別の追加予定がある前提でのみ採用（本ADRの信頼度・再評価条件を参照）。予定が消えれば再評価する。

2. **1操作あたり2行の書き込み**
   - superclass と subtype の2レコードを同一トランザクションで作成する。
   - 対策: 監査ログは状態変更に付随する低頻度の副作用であり、現状の操作頻度では問題にならない。高頻度化したら再評価する。

3. **型別フィルタにジョインが必要**
   - 種別固有項目での絞り込みは subtype テーブルへのジョインを伴う。
   - 対策: 共通の絞り込み（操作者・期間・種別）は `audit_logs` 単独で完結する。型別の詳細絞り込みは頻度が低い。

## 代替案

### 案1: スパム専用の単一テーブル

**概要**: スパム認定専用テーブル1つ（操作者・操作種別・対象・日時）を作る。汎用化しない。

**メリット**:
- 最も単純。1種の要件をそのまま満たす。

**デメリット**:
- 監査種別が増えるたびに同型テーブルを複製するか、後から汎用構造へリファクタする必要がある。
- 操作者横断の監査・共通UIを後付けする際のコストが高い。

**却下理由**: 他の監査種別を追加する具体的予定があるため、専用設計は早晩、複製または作り直しを招く。

### 案2: 単一ポリモーフィックテーブル + JSON ペイロード

**概要**: `audit_logs` 1テーブルに共通項目を持たせ、型別項目は JSON カラムに格納する。subtype テーブルを作らない。

**メリット**:
- テーブルが1つで済み、ペイロードが柔軟。

**デメリット**:
- 型別項目に項目単位の DB 制約（NOT NULL・外部キー）を課しにくく、整合性が弱い（JSON パスのインデックス自体は可能だが、必須・参照制約は失われる）。
- 型別項目での問い合わせが JSON 操作になり、型別の振る舞いをモデルに局所化しにくい。

**却下理由**: 監査ログは整合性が要件。スキーマレスな格納は型別項目の保証を失い、監査用途に不向き。

### 案3: STI（単一テーブル継承）

**概要**: `audit_logs` 1テーブルに `type` 列を持たせ、全種別の項目を同一テーブルに同居させる。

**メリット**:
- Rails ネイティブ。型別モデルを持てる。

**デメリット**:
- 全種別の列が同居し、多くが NULL のスパース・ワイドテーブルになる。型をまたいで NOT NULL を課せない。
- 種別が増えるほど列が肥大化する。

**却下理由**: 型別項目の NOT NULL を課せず、列のスパース化を招く。`delegated_type` は種別別テーブルでこれを回避できる。
