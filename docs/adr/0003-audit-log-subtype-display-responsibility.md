# 0003_監査ログ閲覧における種別別表示責務の配置（Auditable への委譲）

## ステータス

承認済み

## 日付

2026-06-08

## 信頼度

中〜高 — 決定内容と根拠はユーザーが明示的に承認済み。表示多態の前提となる「追加 subtype を複数持つ」ことは ADR-0002 で確認済み（キーワード編集・スパマー登録変更・ユーザーBAN・権限変更等）であり、2番目の委譲先（管理者権限変更監査）の追加が表示パターンを定める分岐点になっている。`delegated_type` は本プロジェクト（Rails 7.2）の安定 API。一方、3種目以降の表示要件はこれからであり、表示責務の最小契約が全種別に無理なく収まるかは運用での検証が残る。

## 再評価条件

- subtype が結局スパム＋権限変更の少数に留まり、表示多態の恩恵が小さい場合（共通契約が単一・少数用途への過剰抽象になる）。
- 表示要件が subtype ごとに大きく分岐し、共通契約（種別ラベル・操作ラベル・対象記述）に収まらず、結局ビュー側の条件分岐が避けられなくなった場合。
- I18n 対応が必要になり、subtype が表示文言（日本語ラベル）を直接持つ方式が運用上問題化した場合（subtype が翻訳キーを返し、ビューが翻訳する形への移行を検討する）。

## コンテキスト

ADR-0002 で汎用 `AuditLog` + `delegated_type` の監査基盤を採用し、最初の委譲先は `SpamModerationAudit` のみだった。その後、汎用の監査ログ閲覧画面（`Admin::AuditLogsController#index` と `app/views/admin/audit_logs/index.html.slim`）が実装された。

いま2番目の委譲先「管理者権限変更監査」（操作種別＝付与/剥奪、対象＝User）を追加する。これにより、複数 subtype をまたぐ表示の多態をどこに配置するかを決める必要が生じた。ADR-0002 はデータモデルのみを決定しており、表示面は未決定である。本ADRはその表示面を補完する。

### 現状の問題点

- 閲覧ビューはスパム認定専用にハードコードされている。種別を固定文字列「スパム認定」で出力し、`detail.action_label` / `detail.target_type` / `detail.target_id` という `SpamModerationAudit` 固有の項目を直接参照している。
- 追加する「管理者権限変更監査」は `AuditLog.recent` に自動的に混入し、現状の閲覧ビューで呼ばれる。しかしこの subtype は `target_type` / `target_id` を持たない（対象は常に User への直接参照）ため、現状ビューのままでは破綻する。
- 本プロジェクトは今後さらに subtype を追加する予定（キーワード編集・スパマー登録変更・ユーザーBAN 等、ADR-0002 に記載）であり、2番目の subtype 追加が表示パターンを定める分岐点になっている。

### 制約条件

- 汎用閲覧ビューは `AuditLog.recent` で全種別を一覧する。種別ごとにビューを分けるのではなく、1つの一覧で多種別を表示する。
- 各 subtype のペイロード構造は異なる（スパム＝対象種別＋ID、権限変更＝User への直接参照）。共通の表示口がなければ、ビューが各種別の内部構造を知る必要が生じる。

### 結合分析

均衡結合フレームワーク（結合の強さ Functional(1)→Integration(4)、距離 同一モジュール(1)→異システム(4)、不均衡＝強さ＞距離 で判定）で評価した。

- 現状: 汎用閲覧ビュー → 各 subtype の固有項目 ＝ Integration(4) × 異コンテキスト(3) ＝ 不均衡（NG）。共有ビューが各種別の内部構造を前提とする典型的な不均衡結合。
- 採用案による改善後: 汎用閲覧ビュー → `Auditable` の共通表示責務 ＝ Functional(1) × 異コンテキスト(3) ＝ 均衡（OK）。

## 決定

各委譲先（`Auditable` を include する subtype）が、表示に必要な共通の表示責務を公開する契約を持つことを決定する。具体的には、**種別ラベル・操作ラベル・対象の表現（人間可読な対象記述）**を各 subtype が自身の責務として提供する。

汎用閲覧ビュー（`app/views/admin/audit_logs/index.html.slim`）はこの抽象的な表示責務のみに依存し、subtype の型（`auditable_type`）による条件分岐や、subtype 固有項目（`target_type` / `target_id` 等）への直接参照を行わない。

### 実装方針

1. 表示に必要な共通契約を `Auditable` 責務として定める。各 subtype（`SpamModerationAudit`、追加する権限変更監査、以降の種別）はこの契約を実装する。
2. 共通契約は方針レベルで次の3点とする。種別ラベル（どの監査か）、操作ラベル（何をしたか）、対象記述（誰/何に対してか、人間可読な短い表現）。対象の表現方法が subtype ごとに異なる差異（スパム＝種別＋ID、権限変更＝User）は、各 subtype が契約の実装内で吸収する。
3. 汎用閲覧ビューは契約のメソッドのみを呼ぶ。型による分岐・固有項目参照を排除し、新しい subtype を追加してもビューは変更不要にする（Open/Closed）。
4. 表示責務の範囲は最小限に限定する。各メソッドは人間可読な短いラベル/対象記述を返すに留め、HTML やビュー固有の整形（リンク化・列レイアウト・I18n の最終整形）はビュー側に残す。

## 結果

### ポジティブな影響

1. **新 subtype 追加時にビュー修正が不要**
   - ビューが `Auditable` 契約のみに依存するため、種別が増えてもビューは変わらない。変更点が subtype 側に局所化される（Open/Closed）。

2. **表示責務の局所化**
   - 各種別の表示文言・対象記述が、その種別を最も理解している subtype モデルに閉じる。型ごとの差異がビューに漏れない。

3. **結合の均衡化**
   - 汎用ビューと subtype の結合が Integration(4) から Functional(1) に下がり、距離（異コンテキスト(3)）との均衡が取れる。共有ビューが各種別の内部構造を知る不均衡を解消する。

### ネガティブな影響・トレードオフ

1. **表示の関心がモデル層に染み出す**
   - ドメインモデル（subtype）が表示用の文言（日本語ラベル等）を持つことになる。
   - 対策: 表示責務を人間可読な短いラベル/対象記述に限定し、HTML やビュー固有の整形はビュー側に残す。I18n が必要になった場合は subtype がキーを返しビューが翻訳する形に発展できる（再評価条件に記載）。

2. **契約の安定性への依存**
   - 全 subtype が同一契約を満たす前提に立つため、表示要件が種別ごとに大きく分岐すると契約が痩せ細るか肥大化する。
   - 対策: 契約を「種別ラベル・操作ラベル・対象記述」の最小3点に絞る。収まらない種別が現れた時点を再評価条件とする。

## 代替案

### 案A: 汎用ビュー側で型による条件分岐（type-switch in view）

**概要**: 汎用ビュー側で `auditable_type` を見て条件分岐し、種別ごとに表示項目を出し分ける。subtype 側に表示責務を持ち込まない。

**メリット**:
- ドメインモデルが表示文言を持たずに済み、表示の関心がモデル層に染み出さない。

**デメリット**:
- subtype を追加するたびにビューを修正する必要があり、ビューが全 subtype の内部構造を知る。
- 結合は Integration(4) × 異コンテキスト(3) のまま不均衡。
- 条件分岐が subtype 数に比例して肥大化し、今後の複数 subtype 追加予定と相性が悪い。

**却下理由**: 追加予定の subtype が複数あり、変更点をビュー1箇所に集約すると拡張のたびに破綻リスクを抱える。結合の不均衡も解消されない。
