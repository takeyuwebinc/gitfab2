# スパム運用機能の拡充 要件定義書

**作成日**: 2026年4月17日
**更新日**: 2026年4月17日

## 1. 機能概要

既存のスパム対策機能を運用面で拡充する。具体的には以下の3点に対応する。

1. スパム投稿の手動対応対象を、既存のコメント2種に加えて「プロジェクトオーナー・コラボレーター以外が投稿可能なコンテンツ」へ拡大する
2. キーワード検出のロジックを強化し、文字分断・全角半角混在・ゼロ幅文字等による迂回に対応する
3. プロジェクトのスパム認定を管理画面から取消可能にする。併せて、あらゆるスパム解除経路でスパマー登録を自動解除する

**主要な提供価値**：
- スパム投稿の手動対応範囲の拡大により、運用で対処できる範囲を広げる
- 既存キーワード登録を活かしつつ、文字分断等の単純な迂回を機械的に検出する
- 誤判定を事後に取り消せるため、運用判断の負荷を下げる

## 2. 機能要件

### 2.1 スパム対応対象の拡大

#### 2.1.1 概要

既存のスパム対応対象は ProjectComment と CardComment の2種類。これに加えて、ログイン済ユーザーであればプロジェクトオーナー・コラボレーター以外でも投稿できる以下のコンテンツを対応対象に追加する。

| 対象 | 内容 | 既存対応 |
|---|---|---|
| ProjectComment | プロジェクトへのコメント | あり |
| CardComment | カードへのコメント | あり |
| Card::Usage | プロジェクトの使い方カード | **なし（追加）** |
| Card::Annotation | State 配下の注釈カード | **なし（追加）** |
| Tag | プロジェクトのタグ | **なし（追加）** |

#### 2.1.2 状態管理

追加対象（Usage、Annotation、Tag）も既存コメントと同じ3状態で管理する。

| 状態 | 説明 |
|---|---|
| unconfirmed | 未確認（初期状態） |
| approved | 承認済 |
| spam | スパム |

#### 2.1.3 スパムマーク操作

システム管理者は管理画面から各対象をスパムとしてマークできる。スパムマーク時、以下が同時に発生する。

1. 対象の status が `spam` に変更される
2. 投稿者が Spammer として登録される（既に登録済の場合は何もしない）
3. 投稿者の未読通知が削除される（既存コメントと同等）

過去に Spammer 登録を解除されたユーザーでも、新たにスパムマークされれば再度 Spammer として登録される。

#### 2.1.4 スパムマーク解除操作

システム管理者は管理画面からスパムマークを解除できる。解除時、以下が発生する。

1. 対象の status が `spam` から `unconfirmed` に変更される
2. 投稿者の Spammer 登録を解除する（詳細は 2.5 参照）

状態遷移の制約：

- `spam` 状態からは `unconfirmed` のみに遷移可能
- `approved` 状態からの直接のスパム解除操作は拒否する（既存コメントと同じ）

#### 2.1.5 表示制御

- 対象の status が `spam` の場合、一般ユーザーの画面には表示されない
- 管理画面では status でフィルターして一覧できる
- 既存コメントのフィルター機能と同じ挙動とする

#### 2.1.6 一括スパムマーク

既存コメントで提供している一括スパムマーク機能と同等の機能を追加対象にも提供する。指定日時以前に投稿された `unconfirmed` 状態のレコードを一括で spam に変更する。

### 2.2 キーワード検出の強化（入力正規化）

#### 2.2.1 概要

既存のキーワード検出機能（`SpamKeywordDetector`）の検出精度を向上させるため、検出対象文字列に対する正規化処理を内部ロジックに追加する。

管理画面のキーワード登録 UI は変更しない。キーワード登録者の作業に影響を与えない形で、検出側のみ強化する。

#### 2.2.2 正規化処理

検出前に以下の正規化を順に適用する。

| # | 正規化 | 対応する迂回 |
|---|---|---|
| 1 | Unicode NFKC 正規化 | 全角/半角混在、合成文字、特殊空白 |
| 2 | ゼロ幅文字の除去 | 目に見えない文字による分断 |
| 3 | 空白・区切り記号の除去 | 文字間に記号・空白を挿入する分断 |
| 4 | 小文字化（既存処理） | 大文字/小文字混在 |

#### 2.2.3 対象と適用タイミング

キーワード照合の実行時に、検出対象（投稿内容）と登録済キーワードの両方を同じ手順で正規化してから照合する。DB に保存されるキーワード本体は未正規化のまま変更しない。既存の登録データに対するマイグレーションは行わない。

#### 2.2.4 除去対象の空白・区切り記号

NFKC 正規化後の文字列から、以下のカテゴリに該当する文字を除去する。

- 空白文字（半角スペース、タブ、改行、NFKC正規化後も残る特殊空白）
- ゼロ幅文字：以下のコードポイントを最低限対象とする
  - U+200B（ZERO WIDTH SPACE）
  - U+200C（ZERO WIDTH NON-JOINER）
  - U+200D（ZERO WIDTH JOINER）
  - U+FEFF（ZERO WIDTH NO-BREAK SPACE）
- 区切りに使われやすい記号：`_` `.` `-` `*` `'` `"` `` ` `` `~` `^` `|` `/` `\` `(` `)` `[` `]` `{` `}` `<` `>`

#### 2.2.5 誤検知への考慮

正規化を強くすると、文字間に記号を挟んだ偶然の一致で誤検知が発生する可能性がある。既存の管理画面から登録済キーワードを個別に無効化（`enabled=false`）する運用で吸収する。

### 2.3 プロジェクトのスパム認定処理の変更

#### 2.3.1 概要

現行のスパム認定処理は関連データの物理削除と匿名化を含むため、取消ができない。取消を可能にするため、スパム認定時の破壊的処理を取りやめる。

#### 2.3.2 変更内容

スパム認定時の処理を以下に変更する。

| 処理 | 変更前 | 変更後 |
|---|---|---|
| プロジェクトの非表示化 | `is_deleted=true` | `is_deleted=true` |
| title / name の匿名化 | あり（'Deleted Project' / ランダム名） | **なし** |
| 関連レコードの物理削除 | likes / states / note_cards / usages / project_comments / figures / tags / collaborations を削除 | **削除しない** |
| オーナーの Spammer 登録 | あり | あり（変更なし） |

#### 2.3.3 表示への影響

`is_deleted=true` のプロジェクトは既存の公開画面のクエリで除外済のため、関連レコードを残しても一般ユーザーには表示されない。管理画面では表示対象となる（詳細は 2.4）。

#### 2.3.4 過去データの扱い

今回の変更は今後のスパム認定から適用する。既にスパム認定済（関連データが物理削除済）のプロジェクトは復元対象外とする。

### 2.4 スパム認定の取消機能

#### 2.4.1 概要

システム管理者はスパム認定済プロジェクトを管理画面から取消できる。

#### 2.4.2 管理画面

- スパム認定済プロジェクト（`is_deleted=true` のプロジェクト）の一覧を管理画面で確認できる
- 一覧から個別プロジェクトのスパム認定を取消できる
- 本施策実施前にスパム認定されたプロジェクト（関連レコードが物理削除済のプロジェクト）も一覧に含まれる

#### 2.4.3 本施策実施前のプロジェクトの扱い

本施策実施前にスパム認定されたプロジェクト（関連レコードが物理削除済、title/name が匿名化済）は、取消しても関連レコード・元のタイトル・元の名称は復元されない。

管理画面の一覧または取消操作時に、当該プロジェクトについては「関連データは復元されない」旨を明示する。

#### 2.4.4 取消時の処理

スパム認定取消時、以下が発生する。

1. プロジェクトの `is_deleted` を `false` に戻す
2. 所有者の Spammer 登録を解除する
   - User 所有の場合: オーナーを対象
   - Group 所有の場合: **取消操作実行時点**で当該 Group に所属している全メンバーを対象（認定時点のメンバー履歴は参照しない）
3. 関連レコード（コメント・カード・タグ等）の status は変更しない。本施策以降の認定分であれば、関連レコードは削除されていないため取消で元通り表示される

#### 2.4.5 関連スパムマークの独立性

プロジェクトのスパム認定取消は、そのプロジェクト配下のコメント・カード・タグに付与されたスパムマークの status を変更しない。

- 個別のスパムマークはプロジェクト単位のスパム認定とは独立に扱う
- 取消後に個別のスパム投稿が残っている場合、それぞれ `Admin::ProjectComments` 等から個別に解除する

### 2.5 スパマー登録の自動解除

#### 2.5.1 概要

あらゆるスパム解除経路で、対象の投稿者の Spammer 登録を自動解除する。

#### 2.5.2 自動解除の発動条件

| 操作 | 解除対象 |
|---|---|
| ProjectComment のスパムマーク解除 | コメント投稿者 |
| CardComment のスパムマーク解除 | コメント投稿者 |
| Card::Usage のスパムマーク解除 | カード作成者 |
| Card::Annotation のスパムマーク解除 | カード作成者 |
| Tag のスパムマーク解除 | タグ作成者 |
| プロジェクトのスパム認定取消 | プロジェクトオーナー（Group の場合は全メンバー） |

#### 2.5.3 解除の単純条件

対象がひとつでもスパム解除されたら、該当ユーザーの Spammer 登録を解除する。他にスパム投稿が残っていても自動解除する。

この方針の運用上の意味：管理者は「このユーザーを Spammer から外したい」と判断した場合、該当ユーザーの投稿のうちどれか1つを解除すればよい。反面、誤って1件を解除すると他のスパム投稿者としての登録も外れるため、管理者は慎重に操作する必要がある。

#### 2.5.4 既存の直接解除経路

既存の `Admin::SpammersController` による Spammer 一覧・直接削除は従来通り残す。

#### 2.5.5 過去データの扱い

本施策の実施前にスパム解除済（status が unconfirmed に戻された）コメント等については、投稿者の Spammer 登録が残っている可能性がある。これらは本機能の対象外とし、必要があれば既存の `Admin::SpammersController` から個別に解除する。

### 2.6 エラー処理

- **URL で指定された対象レコードが存在しない場合**: エラーメッセージを表示する
- **既にスパムマーク済の対象への再マーク**: 冪等に扱い、エラーとしない
- **既に解除済の対象への再解除**: 冪等に扱い、エラーとしない
- **Spammer 登録が既に無いユーザーへの自動解除**: 冪等に扱い、エラーとしない
- **取消対象のプロジェクトが既に `is_deleted=false`**: エラーメッセージを表示する

## 3. 前提条件・制約事項

### 3.1 前提条件

- **システム前提条件**：
  - 既存の SpamCommentable 機構が存在すること
  - 既存の Spammer モデルおよび `Admin::SpammersController` が存在すること
  - 既存の SpamKeywordDetector が存在すること
  - 既存の SpamDesignationService およびプロジェクトのスパム認定機能が存在すること
  - 既存の一括スパムマーク機能（`Admin::Comments::SpamBatchesController` を基底とした派生コントローラ群）が存在すること
  - 既存の管理画面（Admin 機能）が存在すること

### 3.2 制約事項

- **技術的制約**：
  - Ruby 標準の `String#unicode_normalize(:nfkc)` で NFKC 正規化を行うため、外部 Gem の追加は不要
  - 2.2 の入力正規化は検出精度を必ず向上させるものではなく、文字分断・全角半角混在系の迂回にのみ有効
- **対象外**：
  - 過去にスパム認定済（関連データ物理削除済）のプロジェクトの復元
  - 本施策以前にスパム解除済の投稿者に対する Spammer 一括解除
  - 画像内に埋め込まれたキーワードの検出（OCR 等）
  - 同義語・言い換え・URL 難読化への対応
  - 管理画面での正規表現入力
  - 外部スパム判定サービスとの連携
  - 外部委託者向けの権限分離

### 3.3 依存関係

- 既存の Spammer モデル
- 既存の SpamCommentable concern
- 既存の SpamKeywordDetector サービス
- 既存の SpamDesignationService
- 既存の Admin 機能

## 4. 用語定義

| 用語 | 説明 |
|------|------|
| スパム認定 | プロジェクトを `is_deleted=true` にして非表示化する操作（プロジェクト単位） |
| スパム認定取消 | スパム認定を解除し `is_deleted=false` に戻す操作 |
| スパムマーク | コメント・カード・タグの status を `spam` にする操作（個別レコード単位） |
| スパムマーク解除 | `spam` 状態の status を `unconfirmed` に戻す操作 |
| スパマー登録 | 投稿者（User）を Spammer テーブルに登録する操作（ユーザー単位） |
| スパマー登録解除 | Spammer テーブルから対象ユーザーのレコードを削除する操作 |
| 投稿者 | 対象レコード（コメント・カード・タグ）を作成したユーザー |
| 所有者 | プロジェクトのオーナー（User）、または Group 所有の場合は Group に所属する全メンバー |
| 入力正規化 | 検出前に文字列を統一された形式に変換する処理 |
| NFKC 正規化 | Unicode の互換分解後に正準合成する正規化形式。全角/半角、合成文字、特殊空白などを統一できる |
| ゼロ幅文字 | 目に見えない幅を持たない Unicode 文字。U+200B / U+200C / U+200D / U+FEFF 等 |
| 迂回 | スパム投稿者がキーワード検出を避けるために文字装飾や分断を行うこと |

## 5. 受け入れ条件

### 5.1 スパム対応対象の拡大

- [ ] 管理画面から Usage カードをスパムマーク/解除できる
- [ ] 管理画面から Annotation カードをスパムマーク/解除できる
- [ ] 管理画面から Tag をスパムマーク/解除できる
- [ ] Usage / Annotation / Tag の管理画面一覧を status でフィルターできる
- [ ] Usage / Annotation / Tag のスパムマーク時、投稿者が Spammer として登録される
- [ ] スパムマークされた Usage / Annotation / Tag が一般ユーザー画面で非表示になる
- [ ] 指定日時以前の `unconfirmed` 状態のレコードを一括でスパムマークできる
- [ ] 一括スパムマークは `approved` および `spam` 状態のレコードを対象に含めない
- [ ] `spam` 状態から `unconfirmed` への遷移のみ許可される
- [ ] `approved` 状態からの直接のスパム解除操作は拒否される

### 5.2 キーワード検出の強化

- [ ] 全角英数字で登録したキーワードが、半角英数字で投稿された文字列にマッチする
- [ ] `バイアグラ` を登録した場合、`バ_イ_ア_グ_ラ` `バ.イ.ア.グ.ラ` `バ イ ア グ ラ` などの分断投稿が検出される
- [ ] ゼロ幅文字を挿入した投稿が検出される
- [ ] 既存の管理画面キーワード登録 UI に変更がない
- [ ] キーワード照合時、キーワード自体にも同じ正規化が適用される

### 5.3 プロジェクトのスパム認定処理の変更

- [ ] スパム認定時、プロジェクトの title / name が変更されない
- [ ] スパム認定時、likes / states / note_cards / usages / project_comments / figures / tags / collaborations が削除されない
- [ ] スパム認定時、オーナーが Spammer として登録される
- [ ] スパム認定後も一般ユーザー画面には該当プロジェクトが表示されない

### 5.4 スパム認定の取消機能

- [ ] 管理画面でスパム認定済プロジェクト（`is_deleted=true`）の一覧を確認できる
- [ ] 一覧には本施策実施前に認定されたプロジェクトも含まれる
- [ ] 本施策実施前に認定されたプロジェクトについて、関連データが復元されない旨が一覧または取消操作時に明示される
- [ ] 管理画面からスパム認定を取消できる
- [ ] 取消後、プロジェクトが一般ユーザー画面で再び表示される（本施策以降に認定されたプロジェクトの場合）
- [ ] 取消後、関連レコード（コメント・カード・タグ等）が元通り表示される（本施策以降に認定されたプロジェクトの場合）
- [ ] 取消時、User 所有の場合はオーナーの Spammer 登録が解除される
- [ ] 取消時、Group 所有の場合は取消操作時点で Group に所属している全メンバーの Spammer 登録が解除される
- [ ] 取消しても、プロジェクト配下のコメント・カード・タグの status は変更されない

### 5.5 スパマー登録の自動解除

- [ ] コメント（ProjectComment/CardComment）のスパムマーク解除時、投稿者の Spammer 登録が解除される
- [ ] カード（Usage/Annotation）のスパムマーク解除時、投稿者の Spammer 登録が解除される
- [ ] Tag のスパムマーク解除時、作成者の Spammer 登録が解除される
- [ ] プロジェクトのスパム認定取消時、所有者の Spammer 登録が解除される
- [ ] ユーザーが複数のスパム投稿で登録されている場合でも、どれか1つの解除で Spammer 登録が外れる
- [ ] Spammer 登録を解除されたユーザーが再度スパムマークされた場合、Spammer として再登録される
- [ ] Spammer 登録が既に無いユーザーに対する自動解除操作でエラーが発生しない
- [ ] 既存の `Admin::SpammersController` による直接削除機能が引き続き動作する

## 6. 今後の拡張予定（対象外）

以下は本要件の対象外とする。キーワード検出の強化（2.2）は「現時点で必要な最小対応」であり、以下は別途の詳細調査で検討する。

- 画像内キーワードの検出（OCR 等）
- 同義語・言い換え検出
- URL 難読化対応
- 管理画面での正規表現入力
- 外部スパム判定サービス（Akismet 等）との連携
- 過去にスパム認定済（関連データ物理削除済）のプロジェクトの復元
- 本施策前にスパム解除済の投稿者に対する Spammer 一括解除
- 外部委託者向けの権限分離
