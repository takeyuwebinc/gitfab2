# 管理者権限管理機能 要件定義書

## 1. 機能概要

現在、管理者権限は `users.authority='admin'` の手作業 UPDATE で付与している。これまで運用上問題がなかったのは、管理者権限を他者に付与する状況が発生していなかったため。今後、管理業務を他者に任せる可能性があるため、データベースの直接操作以外の手段で権限を管理できるようにする。

本機能では、管理画面からユーザーを検索・選択し、管理者権限（`authority='admin'` か否か）を切り替えられるようにする。

**主要な提供価値**：
- データベースを直接操作せずに管理者権限を付与・剥奪できる
- 誰がいつ誰の管理者権限を変更したかを後から確認できる
- 誤操作（自分自身や最後の管理者の権限剥奪）による管理不能状態を防止する

## 2. 機能要件

### 2.1 基本処理フロー

1. **ユーザーの検索・一覧表示**
   - システム管理者が管理画面のユーザー一覧でユーザー名・メールアドレスを条件に検索する
   - 各ユーザーについて、管理者権限の有無を一覧で表示する

2. **管理者権限の付与・剥奪**
   - システム管理者が個別ユーザーの管理者権限を ON（付与）／OFF（剥奪）に切り替える
   - 切り替え結果は対象ユーザーの管理者権限の有無に即時反映される

3. **誤操作防止チェック**
   - 自分自身の管理者権限を OFF にする操作は拒否する
   - システム上で最後の1名となる管理者の権限を OFF にする操作は拒否する

4. **変更の記録**
   - 権限が実際に変更された場合、その操作を監査ログとして記録する（詳細は4章）

### 2.2 データ仕様

- **データソース**：
  - 既存の `users` テーブル（`authority` カラム）をそのまま使用する
  - 監査ログ用に新規テーブルを追加する（4章参照）
- **データ取得要件**：
  - ユーザーをユーザー名・メールアドレスで検索する
  - 一覧表示項目：ユーザーの識別情報、管理者権限の有無
- **データ更新要件**：
  - 対象ユーザーの管理者権限の有無（`authority='admin'` か否か）を更新する

### 2.3 エラー処理

- **エラー条件**：
  - 自分自身の管理者権限を OFF にしようとした：操作を拒否し、理由を表示する
  - 最後の1名となる管理者の権限を OFF にしようとした：操作を拒否し、理由を表示する
  - 管理者でないユーザーがこの画面にアクセスした：既存の管理画面と同じ挙動（アクセス不可）とする
- **エラー通知**：
  - 拒否した操作については、なぜ実行できないかを操作者に画面上で示す

## 3. 前提条件・制約事項

### 3.1 前提条件

- **システム前提条件**：
  - 既存の管理画面の認可は「ログイン済み、かつ `is_system_admin?`（`authority == 'admin'`）」のみで判定している
  - 管理画面配下に「ユーザー一覧」は現状存在しない
  - システム管理者は、通常のユーザーアカウントに `authority='admin'` が立っているだけである
- **データ前提条件**：
  - `authority` カラムは文字列で、実運用上の値は `'admin'`（管理者）または未設定（一般ユーザー）の二値である

### 3.2 制約事項

- **権限モデルの制約**：
  - 認可は既存どおり「管理者か否か」の二値とする
  - 管理者権限を付与された者は、全管理機能（コメント削除、プロジェクトのスパム認定、スパムキーワード登録、他ユーザーの管理者権限変更などを含む）の全権限を持つ
  - 既存の管理画面の認可ロジック（`is_system_admin?` による判定）は変更しない
- **業務上の制約**：
  - 本機能は、管理者を任せる相手が信頼でき、利用できる範囲を制限する必要がない場合を前提とする

### 3.3 依存関係

- 既存の `User` モデルおよび `authority` カラム
- 既存の管理画面の認可機構（`is_system_admin?`）

## 4. 監査・記録要件

### 4.1 記録の目的

- 業務追跡・不正検知：誰がいつ誰の管理者権限を付与・剥奪したかを後から確認できるようにする
- 根拠：法令・契約上の記録義務は現時点で明確な根拠なし。社内の運用上の必要性に基づく

### 4.2 対象操作

- 記録対象：管理者権限の付与（OFF→ON）および剥奪（ON→OFF）
- 一覧の閲覧・検索などの読み取り操作は対象外とする
- 誤操作防止によりブロックされた操作（自分自身の OFF、最後の1名の OFF）は、権限が実際に変更されないため記録対象外とする

### 4.3 記録すべき情報

- 操作日時
- 操作者（誰が変更したか）
- 対象ユーザー（誰の権限を変更したか）
- 操作種別（付与／剥奪）

詳細な記録スキーマは機能設計書で定義する。

### 4.4 保存期間

- 無期限保存とする
- 根拠：管理者権限の変更は低頻度で記録件数が少なく、保存コストが低いため。法定保存期間の定めはない

### 4.5 参照者

- 参照者：システム管理者
- 参照場面：現在および過去の管理者権限保有状況の確認、不正・トラブル発生時の経緯調査
- 記録の閲覧 UI は設けず、記録のみを行う

### 4.6 PII（個人情報）の取り扱い

- 記録に含まれるのは、操作者・対象ユーザーの識別情報のみ
- 参照者がシステム管理者に限られるため、マスキング・匿名化は行わずそのまま記録する
- 本人開示請求への対応は今回想定しない

## 5. 用語定義

| 用語 | 説明 |
|------|------|
| 管理者 / システム管理者 | `authority='admin'` を持つユーザー。全管理機能を利用できる |
| 管理者権限 | 管理画面の全機能を利用できる権限。本システムでは `authority='admin'` の有無で表される二値 |
| 監査ログ | 管理者権限の変更操作（付与・剥奪）の記録 |

## 6. 受け入れ条件

- [ ] ユーザー一覧でユーザー名・メールアドレスを条件に検索できる
- [ ] 各ユーザーの管理者権限の有無が一覧で表示される
- [ ] 個別ユーザーの管理者権限を ON／OFF に切り替えられる
- [ ] 自分自身の管理者権限を OFF にしようとすると拒否される
- [ ] 最後の1名となる管理者の権限を OFF にしようとすると拒否される
- [ ] 権限が変更された操作が監査ログに記録される（操作者・対象・操作種別・日時）
- [ ] 管理者以外はこの画面を利用できない
