# 管理者権限管理機能 機能設計書

**機能名**: 管理者権限管理機能
**要件定義書**: [管理者権限管理機能 要件定義書](../requirements/admin-permissions-management.md)

> **設計方針**
>
> この機能設計書は、実装の詳細に影響されないコアドメインの設計を記載する。
> 新規に作るものは責務レベルで記述し、設計時点で既に存在するもの（既存テーブル・
> 既存モデル・既存コントローラ等）は事実として具体名で記載する。

## 1. 機能概要

### 1.1 目的

データベースの直接 UPDATE に代わり、管理画面から管理者権限（`users.authority` が `'admin'` か否か）を付与・剥奪できるようにする。

- データベースを直接操作せずに管理者権限を付与・剥奪できる
- 誰がいつ誰の管理者権限を変更したかを後から確認できる
- 誤操作（自分自身・最後の1名の剥奪）による管理不能状態を防止する

### 1.2 主要機能

1. **ユーザー検索・一覧表示**: ユーザー名・メールアドレスで検索し、各ユーザーの管理者権限の有無を一覧表示する
2. **管理者権限の付与・剥奪**: 個別ユーザーの管理者権限を ON（付与）／OFF（剥奪）に切り替える
3. **誤操作防止**: 自分自身の剥奪、および最後の1名となる管理者の剥奪を拒否する
4. **変更の監査記録**: 権限が実際に変更された場合のみ、操作を監査ログに記録する

### 1.3 処理フロー概要

#### 1.3.1 ユーザー検索・一覧表示

1. システム管理者が管理画面のユーザー一覧を開く
2. ユーザー名・メールアドレスを条件に検索する（部分一致、未指定時は全件）
3. 各ユーザーの識別情報と管理者権限の有無を一覧で表示する（ページ分割あり）

#### 1.3.2 管理者権限の付与・剥奪

4. 対象ユーザーと操作（付与／剥奪）を受け取る
5. 誤操作防止チェックを実施する（1.3.3）
6. 拒否対象の場合、理由を画面に表示し、権限は変更せず、監査記録も行わない
7. 操作が現在の状態と同じ場合（既に管理者への付与、既に一般への剥奪）は、状態を変更せず成功扱いとし、監査記録も行わない（冪等。エラーにはしない）
8. 状態が変化する場合、対象ユーザーの管理者権限を更新する（付与＝`authority` を `'admin'` に、剥奪＝未設定に）
9. 権限が実際に変化した場合に限り、状態変更と同一トランザクション内で監査ログを記録する（7章）
10. 処理後、ユーザー一覧に最新の管理者権限の有無を反映して表示する（要件2.2「即時反映」）

#### 1.3.3 誤操作防止チェック

11. 操作者自身を対象とする剥奪は拒否する
12. 剥奪後に管理者が0名になる剥奪（最後の1名の剥奪）は拒否する

## 2. 設計判断

### 2.1 採用した実装パターン

| # | 判断ポイント | 採用案 | 関連ADR |

| 2 | 監査ログ閲覧における種別別の表示 | 各委譲先（`Auditable`）が共通の表示責務を公開し、汎用閲覧ビューは抽象責務のみに依存 | [ADR-0003](../adr/0003-audit-log-subtype-display-responsibility.md) |
| 3 | 監査記録のトリガー | 既存の監査記録パターン（`SpamModerationAuditable`）を踏襲。`Current.admin` が存在し、かつ状態が実際に変化した時のみ、同一トランザクションで記録 | （既存パターン踏襲のため新規ADRなし） |

判断ポイント3は既存パターンの踏襲であり、複数案比較を省略した。

### 2.2 結合評価サマリ

均衡結合フレームワーク（強さ Functional(1)→Integration(4) × 距離 同一モジュール(1)→異システム(4)、不均衡＝強さ＞距離）で評価した。

| # | 起点 → 終点 | 強さ | 距離 | 均衡 | 備考 |
|---|-----------|------|------|------|------|
| 1 | 権限変更操作（新規）→ `User#authority`（既存） | Model(2) | 同一コンテキスト(2) | OK | 既存の管理画面が core モデルを直接操作する慣習に整合 |
| 2 | 権限変更監査記録（新規）→ `AuditLog`/`Auditable` 基盤（既存） | Contract(3) | 異コンテキスト(3) | OK | ADR-0002 が用意した拡張シーム（`delegated_type` 契約）に従う |
| 3 | 汎用閲覧ビュー（既存）→ `Auditable` の共通表示責務（改善後） | Functional(1) | 異コンテキスト(3) | OK | ADR-0003 で表示責務を各 subtype に局所化した結果 |

**許容している不均衡**: なし。

**設計改善の経緯**: 改善前は、汎用閲覧ビューが各 subtype の固有項目（`action_label`/`target_type`/`target_id`）を直接参照し、種別を固定文字列でハードコードしていた（Integration(4) × 異コンテキスト(3) ＝ 不均衡）。本機能で2番目の subtype を追加するにあたり、表示責務を `Auditable` 契約として各 subtype に持たせる設計（ADR-0003）に改め、Functional(1) × 異コンテキスト(3) ＝ 均衡 に解消した。

## 3. データ要件

### 3.1 既存データの利用

#### 3.1.1 ユーザー

**テーブル名**: `users`（既存）

管理者権限は既存の `authority` カラム（文字列）で表される。実運用上の値は `'admin'`（管理者）または未設定（一般ユーザー）の二値。読み取り判定は既存の `User#is_system_admin?`（`authority == 'admin'`）を用いる。本機能はこの `authority` を更新対象とし、カラム追加・型変更は行わない。

本機能が参照・更新する既存カラムは次のとおり。

| 項目名 | 用途 | 説明 |
|--------|------|------|
| name | 検索条件・一覧表示 | ユーザー名（部分一致検索の対象） |
| email | 検索条件・一覧表示 | メールアドレス（部分一致検索の対象） |
| authority | 一覧表示・更新対象 | 管理者権限の有無。`'admin'` か未設定の二値 |

### 3.2 監査ログ

監査ログは ADR-0002 の構造（共通メタデータの委譲元＋種別別の委譲先）に従う。本機能では新しい委譲先（権限変更監査）を追加する。

#### 3.2.1 監査ログ共通メタデータ（既存）

**テーブル名**: `audit_logs`（既存）

操作者・発生日時・操作元 IP アドレスを保持する委譲元。本機能の権限変更監査も、この共通メタデータを利用する。新しい委譲先を `delegated_type` の対象種別に追加する。

#### 3.2.2 権限変更監査（新規・委譲先）

**責務**: 管理者権限の変更（付与／剥奪）の種別別詳細を保持する。`Auditable` を備え、委譲元 `audit_logs` と1対1で結びつく。`spam_moderation_audits` と並列の委譲先。

**テーブル名**: `admin_authority_audits`

| 項目名 | キー | データ型 | 必須 | 説明 |
|--------|------|----------|------|------|
| id | PK | bigint | ○ | 主キー |
| action | | integer | ○ | 操作種別（0: 付与, 1: 剥奪） |
| target_user_id | IDX | bigint | ○ | 対象ユーザー。`users` への直接参照だが緩い参照（FK制約は張らない） |
| created_at | | datetime | ○ | 作成日時 |
| updated_at | | datetime | ○ | 更新日時 |

**制約・方針**:
- `target_user_id` には INDEX を付与する。
- 対象は常にユーザーであるため、ポリモーフィックではなくユーザーへの直接参照とする。
- FK 制約は張らない（緩い参照）。対象ユーザーが削除されても監査記録は残す（`SpamModerationAudit` の「対象が削除されても監査記録は残す」方針と整合）。
- 操作者・発生日時・操作元 IP は委譲元 `audit_logs` が保持する（本テーブルには持たない）。

**データ作成タイミング**:
- 管理者権限が実際に変化した時（付与: 一般→管理者、剥奪: 管理者→一般）に、状態変更と同一トランザクションで委譲元 `AuditLog` と本委譲先の1組を作成する。

## 4. 計算ロジック

### 4.1 最後の1名判定

剥奪を許可してよいかは、剥奪後に管理者が残るかで判定する。

```
管理者数 = authority が 'admin' であるユーザーの件数

剥奪許可条件 = （対象が現在管理者である）かつ（管理者数 − 1 ≥ 1）
```

判定手順:
1. 対象ユーザーが現在管理者でなければ、剥奪は状態変化を伴わない（記録対象外、4.2 と合わせて扱う）。
2. 対象を除いた管理者が1名以上残るなら剥奪を許可する。残らない（最後の1名）なら拒否する。
3. 同時実行で複数の最後の管理者が同時に剥奪され0名化する競合を防ぐため、件数確認と更新は同一トランザクション内で行い、対象更新時に悲観的ロックで原子性を確保する（5.2.3）。

## 5. ビジネスルール

### 5.1 認可

#### 5.1.1 アクセス制御

- **ルール**: 本画面の利用は、ログイン済みかつ管理者（`is_system_admin?`）に限る。
- **実現**: 既存の `Admin::ApplicationController` の認可（`is_system_admin?` 判定、不適合時は `root_path` へリダイレクト）を継承する。認可ロジックは変更しない。
- **副次効果**: 同コントローラは認可済みの一点で `Current.admin`（操作者）と `Current.ip_address`（操作元 IP）を設定する。監査記録はこれを参照する。

### 5.2 誤操作防止

#### 5.2.1 自分自身の剥奪禁止

- **ルール**: 操作者自身を対象とする剥奪は拒否する。
- **目的**: 操作者が自らを管理不能状態にすることを防ぐ。
- **挙動**: 権限は変更せず、拒否理由を画面に表示する。監査記録は行わない。

#### 5.2.2 最後の1名の剥奪禁止

- **ルール**: 剥奪後に管理者が0名になる剥奪を拒否する（4.1）。
- **目的**: システムから管理者が消失することを防ぐ。
- **挙動**: 権限は変更せず、拒否理由を画面に表示する。監査記録は行わない。

#### 5.2.3 同時実行時の整合性

- **ルール**: 「最後の1名」判定と権限更新は原子的に行う。
- **目的**: 複数管理者が同時に剥奪し、判定をすり抜けて0名化することを防ぐ。
- **挙動**: 件数確認・更新・監査記録を同一トランザクション内で実行し、対象更新時に悲観的ロックを用いる。

### 5.3 記録対象の限定

- **ルール**: 監査記録は、権限が実際に変化した付与（一般→管理者）・剥奪（管理者→一般）のみを対象とする。
- **対象外**:
  - 誤操作防止で拒否された操作（変化が起きないため）
  - 状態変化を伴わない操作（既に管理者への付与、既に一般への剥奪）
  - 一覧の閲覧・検索などの読み取り操作

## 6. 画面表示機能

### 6.1 画面構成

本機能は管理画面（`admin` レイアウト）配下に新設するユーザー一覧画面で提供する。

#### 6.1.1 ユーザー一覧画面

- **検索領域**: ユーザー名・メールアドレスを条件とする検索フォーム（部分一致、未指定時は全件）。検索は既存の管理画面慣習（`params[:q]` による絞り込み）に倣う。
- **一覧領域**: 各ユーザーの識別情報（ユーザー名・メールアドレス等）と管理者権限の有無を表示する。件数が多いためページ分割する。
- **操作領域**: 各ユーザー行に、管理者権限の付与／剥奪を切り替える操作を配置する。

#### 6.1.2 監査ログ閲覧画面への反映（既存画面の拡張）

- 本機能専用の閲覧 UI は設けない。ただし、権限変更監査は既存の汎用監査ログ閲覧画面（`app/views/admin/audit_logs/index.html.slim`）に自動的に表示される。
- 当該ビューは現状スパム認定専用にハードコードされているため、ADR-0003 に従い、各 subtype が公開する表示責務（種別ラベル・操作ラベル・対象記述）に依存する形へ改める。
- 列構成は、現行の「種別／操作／対象種別／対象ID」を「種別／操作／対象」に一般化する（対象種別＋対象IDの2列を、各 subtype が返す対象記述の1列に集約）。
- 種別ラベル・操作ラベル・対象記述は、保存項目ではなく各 subtype が表示責務として算出する導出値である（データ要件3章には含めない）。
- この既存画面の改修は、要件定義書4.5「閲覧UIを設けない」の作成後に汎用閲覧画面が実装された経緯を踏まえた設計判断であり、ADR-0003 に基づく。要件には明示根拠を持たない設計由来の変更である点を明記する。

### 6.2 エラーハンドリング

#### 6.2.1 エラー分類

| エラーレベル | 説明 | 処理継続 |
|-------------|------|----------|
| ERROR | 自分自身の剥奪、最後の1名の剥奪 | 中止（権限変更・監査記録を行わず、拒否理由を表示） |
| ERROR | 管理者以外のアクセス | 中止（既存の管理画面と同じ挙動でアクセス不可） |

#### 6.2.2 拒否時の通知

- 拒否された操作については、なぜ実行できないか（自分自身は剥奪できない／最後の管理者は剥奪できない）を操作者に画面上で示す。

## 7. ログ設計

要件定義書 4章「監査・記録要件」に対応する。本機能の「監査ログ」は、ログストリームではなくデータベース上の監査レコード（ADR-0002 の `AuditLog` 基盤）として実現する。

### 7.1 ログカテゴリ

| カテゴリ | 用途 | 想定保存期間 |
|---------|------|-------------|
| 監査ログ | 管理者権限変更の説明責任・不正監査（誰がいつ誰の権限を変更したか） | 無期限（要件4.4。低頻度・少件数で保存コストが低く、法定保存期間の定めもない） |

業務イベントログ・アクセスログ・アプリケーションログ・エラーログは本機能で新設しない（既存方針に従う）。

### 7.2 監査ログの記録項目

要件定義書 4.3 に対応する。共通メタデータは委譲元 `audit_logs`、種別別項目は委譲先 `admin_authority_audits` が保持する。

| 項目 | 保持先 | 説明 | 必須 |
|------|--------|------|------|
| 操作日時 | audit_logs | 記録の発生日時 | ○ |
| 操作者 | audit_logs | 権限を変更したシステム管理者（`operator`） | ○ |
| 操作元IPアドレス | audit_logs | 操作者のリクエスト元 IP | 任意 |
| 操作種別 | admin_authority_audits | 付与／剥奪 | ○ |
| 対象ユーザー | admin_authority_audits | 権限を変更された対象（`target_user`） | ○ |

**用語の対応**: 「操作者」は操作を行うシステム管理者（`Current.admin`、データ上は委譲元 `audit_logs` の `operator`）を指す。「対象ユーザー」は権限を変更されるユーザー（`admin_authority_audits` の `target_user`）を指す。要件定義書5章の「管理者／システム管理者」は本設計の「操作者」「管理者」に対応する。

### 7.3 記録のトリガー

- 既存の `SpamModerationAuditable` パターンを踏襲する。
- `Current.admin`（操作者）が存在し、かつ管理者権限が実際に変化した場合のみ記録する。操作者を持たない経路（ジョブ・rake 等）は自然に対象外となる。
- 状態変更と同一トランザクション内で記録し、記録失敗時は権限変更ごとロールバックする。

### 7.4 PIIマスキング方針

| 項目 | マスキングパターン | 理由 |
|------|------------------|------|
| 操作者・対象ユーザーの識別情報 | そのまま記録 | 参照者がシステム管理者に限られ、識別情報のみで本文等の機微情報を含まないため（要件4.6）。匿名化・マスキングは行わない。 |

### 7.5 出力先・保存期間

| カテゴリ | 出力先（概念） | 保存期間 |
|---------|---------------|---------|
| 監査ログ | 監査ログ用データストア（既存 `audit_logs` ＋ `admin_authority_audits`） | 無期限（自動削除しない） |

### 7.6 参照

- 記録の参照者はシステム管理者。参照場面は管理者権限保有状況の確認、不正・トラブル時の経緯調査。
- 参照は既存の汎用監査ログ閲覧画面で行う（6.1.2）。本機能専用の閲覧 UI は新設しない。
